## Synthese executive
Le projet repose sur une base moderne et globalement saine (React + TS + Supabase + RLS), mais presente trois risques structurants majeurs:
1. **Duplication du code applicatif** entre la racine et `dream-canvas` (risque eleve de divergence et de regressions).
2. **Maturite qualite insuffisante** (quasi absence de tests metier, pas de pipeline CI visible).
3. **Risque cout/abuse IA** sur les fonctions serveur, en particulier `generate-panel-image` sans quota explicite.

## Constat principal: structure en double
Le depot contient deux arborescences applicatives quasi miroirs:
- `src`, `supabase`, `Produit`, etc. a la racine
- `dream-canvas/src`, `dream-canvas/supabase`, `dream-canvas/Produit`, etc.

### Risque
- Source de verite ambigue
- Correctifs appliques dans un arbre mais oublies dans l'autre
- Difficulte de revue de PR et de release
- Dette de maintenance elevee

### Recommandation P0
- Declarer un **seul** repertoire de reference (ex: racine)
- Geler l'autre copie (archive) puis suppression planifiee
- Documenter la decision dans `README.md`

## Architecture fonctionnelle (etat observe)
- Authentification et sessions via `src/hooks/useAuth.tsx` et `ProtectedRoute`
- Parcours principal:
  - `Projects` -> `ProjectDetail` -> `ChapterDetail`
  - `ScenarioSection` (generation scenario IA + editions chapitres scenario)
  - `EditionSection` / `Panel` (composition visuelle)
  - `AssetLibrary` (generation et gestion d'assets)
- Couplage fort frontend <-> Supabase (tables + Edge Functions)

### Zones a forte complexite
- `src/pages/ChapterDetail.tsx` (volume et logique mixte UI/metier/persistance)
- `src/components/project/ScenarioSection.tsx` (plusieurs modes IA, orchestration riche)
- `src/services/panels.ts` (logique layout + persistance + generation image)

## Audit qualite code et tests

### Points positifs
- Stack recente (`vite`, `react`, `typescript`, `vitest`, `eslint`)
- Separation partielle en `hooks` + `services`
- Presence de typage et utilitaires partages

### Faiblesses majeures
- Couverture tests tres faible (1 test exemple detecte dans `src/test/example.test.ts`)
- Pas de CI visible (`.github/workflows` absent)
- Pas de script `typecheck` dedie dans `package.json`
- Contradiction de configuration TypeScript:
  - `tsconfig.app.json` en strict
  - `tsconfig.json` relache plusieurs garde-fous (`allowJs`, `strictNullChecks: false`, `noImplicitAny: false`)
- Multiples assertions dangereuses/casts:
  - Occurrences de `user!.id` dans plusieurs hooks
  - Occurrences de `as any` / `as unknown as` dans des zones critiques

### Impact
- Regressions fonctionnelles probables lors des evolutions rapides
- Bugs runtime possibles mal couverts (auth, IA, edition de panels)
- Signal qualite insuffisant avant livraison

## Audit securite (Supabase + Edge Functions)

### Points solides
- RLS activee sur les tables principales
- Policies de base coherentement orientees ownership par `auth.uid() = user_id`
- Verification JWT et ownership dans les Edge Functions critiques

### Risques identifies
- **P0 - Cout/abuse**: `generate-panel-image` sans gouvernance quota comparable a `generate-asset-image`
- **P1 - Surface d'exposition**: `verify_jwt = false` sur les fonctions (compense par verif applicative, mais fragile a l'echelle)
- **P1 - CORS permissif**: fallback `*` observe dans les fonctions (a restreindre en production)
- **P1 - Confidentialite potentielle**: policy storage de lecture large selon le mode d'exposition bucket
- **P2 - Robustesse migration**: certaines migrations peu idempotentes

### Recommandations securite prioritaires
1. Aligner la logique quota et tracking usage sur toutes les fonctions de generation image.
2. Introduire un mecanisme atomique de consommation quota (RPC SQL transactionnelle).
3. Rendre `ALLOWED_ORIGIN` stricte en production (allowlist).
4. Revalider les policies storage selon besoin reel (public vs prive utilisateur).

## Audit performance et cout

### Risques
- Appels IA potentiellement couteux (tokens elevables, prompts longs, absence de limites strictes)
- Parcours de reordonnancement a updates multiples non transactionnels
- Gros composants monolithiques susceptibles de rerenders couteux
- Indexation DB a renforcer sur certaines colonnes de filtrage frequentes

### Recommandations
- Ajouter limites strictes de taille/longueur sur inputs IA
- Ajouter instrumentation de latence/erreur pour chaque edge function
- Revoir les operations de reorder avec strategie transactionnelle
- Ajouter indexes sur relations et filtres les plus frequents (`user_id`, `project_id`, `chapter_id`)

En traitant ces axes dans l'ordre recommande (P0 -> P1 -> P2), la velocite de livraison restera elevee tout en diminuant fortement les regressions et incidents.

## Synthèse exécutive

- **Produit** : DreamWeave est une application web pour créer des webtoons/BD assistés par IA, avec un front React/Vite et un backend géré par Supabase (Postgres, Auth, Storage, Edge Functions).
- **Maturité technique** : architecture front propre (services + hooks + types partagés), intégration Supabase avancée (Edge Functions, quotas d’usage, plans Free/Pro), UI moderne (Tailwind + shadcn-ui). La base de code est globalement saine et bien typée.
- **Principales faiblesses** : très peu de tests automatisés (un seul test de démo), absence visible de CI, quelques relâchements TypeScript au niveau racine, dépendance forte à des Edge Functions complexes sans tests, et certains aspects sécurité/performances qui reposent sur la bonne configuration Supabase hors repo.
- **Priorités** : mettre en place une CI simple (lint + tests), étoffer les tests (services/hook critiques + Edge Functions), clarifier/renforcer les règles RLS sur toutes les tables, et durcir un peu les règles TS/ESLint autour des valeurs inutilisées et des null/undefined.

## Frontend

### Stack & architecture

- **Stack principale**
  - React 18 + Vite 7 (`package.json`, `vite.config.ts`)
  - TypeScript 5 (`typescript` devDep)
  - React Router DOM v6 (`react-router-dom`) pour le routing SPA
  - React Query v5 (`@tanstack/react-query`) pour la gestion serveur/state async
  - Supabase JS v2 (`@supabase/supabase-js`) pour Auth, PostgREST, Storage
  - shadcn-ui / Radix UI + Tailwind CSS pour la couche UI
  - Vitest + Testing Library pour les tests (mais très peu utilisés)

- **Structure par dossiers**
  - **`src\App.tsx`** : composition racine (QueryClientProvider, Router, AuthProvider, Suspense, Toasters) et définition des routes (Landing, Auth, Dashboard, Projects, ProjectDetail, Profile, Plans, ChapterDetail, NotFound).
  - **`src\main.tsx`** : bootstrap ReactDOM + initialisation du thème (dark mode via `localStorage` + `prefers-color-scheme`).
  - **`src\pages\*.tsx`** : pages fonctionnelles (Accueil, Auth, Dashboard, Projects, ProjectDetail, ChapterDetail, Plans, Profile, etc.).
  - **`src\components\DashboardLayout.tsx` + composants project/** : layout authentifié et écrans métier (édition panels, asset library, etc.).
  - **`src\components\ui\*.tsx`** : bibliothèque de composants UI (button, dialog, input, table, etc., style shadcn).
  - **`src\hooks\*.ts(x)`** : hooks métiers (auth, projets, assets, chapters, IA scénario, panels…).
  - **`src\services\*.ts`** : couche d’accès aux données (Supabase) et aux Edge Functions (appel HTTP externe).
  - **`src\types\index.ts`** : types métier (Projects, Assets, Panels, Tiers, etc.) dérivés du schéma Supabase.
  - **`src\integrations\supabase\*`** : client Supabase configuré + types générés.

### Patterns principaux

- **Séparation claire responsabilités**
  - **Services** : `projects.ts`, `assets.ts`, `scenarioAI.ts`, `panels.ts`, etc. encapsulent Supabase + Edge Functions, avec types explicites (ex. `fetchProjects(): Promise<Project[]>`).
  - **Hooks React Query** : `useProjects`, `usePanels`, `useScenarioAI`, `useAssets`, `useUserPlan` encapsulent la logique de requêtage, de cache et d’invalidation (via `useQuery`/`useMutation`).
  - **Contexte Auth** : `useAuth` expose `user`, `session`, `loading` + méthodes (`signUp`, `signIn`, `signInWithGoogle`, `resetPassword`, `updatePassword`, `signOut`).
  - **Types centralisés** : `src\types\index.ts` s’appuie sur `Tables`, `TablesInsert`, `TablesUpdate`, `Enums` générés pour la cohérence BD ↔ front.

- **Routing & lazy loading**
  - Pages importées en lazy (`lazy(() => import("./pages/Landing"))`, etc.) avec `Suspense fallback` personnalisé (`PageLoader`).
  - Route protégée via `ProtectedRoute` (non inspecté mais utilisé dans toutes les routes /dashboard/*).

- **UI & design system**
  - Tailwind configuré via `tailwind.config.ts` avec thème riche (couleurs CSS vars, animations nombreuses, typographies `Quicksand` & `Nunito`, variantes `sidebar`, `lavender/peach/mint`, etc.).
  - Composants UI factorisés dans `components\ui\` (pattern shadcn), ce qui centralise l’accessibilité et les styles.

### Points forts

- **Architecture front claire et modulaire**
  - Découplage data (services) / état asynchrone (hooks React Query) / présentation (composants, pages).
  - Types métier alignés sur le schéma Supabase, évitant les divergences manuelles.

- **Gestion de l’auth robuste côté UI**
  - `useAuth` :
    - Gestion fine des erreurs Supabase (codes et messages spécifiques : `USER_ALREADY_EXISTS`, `EMAIL_NOT_CONFIRMED`, etc.).

- **Expérience utilisateur travaillée**
  - Pages `Landing` et `Dashboard` très soignées, animations `framer-motion`, badges de plan, barre de progression d’usage.
  - Lazy loading des pages avec écran de chargement dédié, réduisant le temps inter-écran perçu.
  - Thème dark/clair persistant, initialisé tôt dans `main.tsx`.

- **Cohérence typage et domaine**

  - Modélisation précise du layout de panels (`PanelBlock`, `PanelLayout`, `ColorBlock`, `SpeechBubble`), alignée sur les colonnes JSONB en base.

### Infrastructure & configuration

- **Projet Supabase**
  - `supabase\config.toml` :

    - `verify_jwt = false` pour les fonctions :
      - `generate-asset-image`

- **Fonctions Edge**
  - `generate-scenario-ai` : appelle Groq (Llama 3.3 70B) pour générer/sculpter scénario/chapitre ou découper en panels.
  - `generate-asset-image` : appelle FAL.ai (FLUX Schnell/Flux-2 Pro/Edit) pour générer des assets avec gestion de plans Free/Pro et quotas mensuels.
  - `generate-panel-image` : génère des images par bloc de panel, en stockant les résultats dans Supabase Storage.

### Schéma & migrations

- **Tables principales (via `src\integrations\supabase\types.ts`)**
  - `projects` : projet de webtoon (titre, description, style, `panels_target_per_chapter`, `style_image_urls`, `cover_url`, `user_id`, timestamps).
  - `chapters` : chapitres liés à un projet (`chapter_number`, `linked_scenario_chapter_id`, synopsis, user_id, etc.).
  - `panels` : panels d’un chapitre, avec colonnes JSONB pour `layout`, `color_blocks`, `speech_bubbles`, `transition_effects`, `motion_lines`, etc.
  - `assets` : assets graphiques (type enum `asset_type` : `character` | `background` | `object`, diverses URLs d’images, `metadata`, user_id, project_id).
  - `profiles` : profil utilisateur (`display_name`, `avatar_url`, `plan` free/pro).
  - `usage` : journal d’usage pour le rate limiting (user_id, action, created_at).
  - `scenario_chapters` / `scenario_versions` : gestion de versions de scénario, contenu texte, outline panels, etc.

- **Migrations notables**
  - `20260215200000_edition_oeuvre_chapters_panels.sql`
    - Ajout `linked_scenario_chapter_id` sur `chapters` avec index filtré (`IS NOT NULL`) pour relier un chapitre éditorial à un chapitre de scénario.
    - Ajout de `layout`, `transition_effects`, `motion_lines` (JSONB) sur `panels`, avec commentaires explicites sur leur rôle.
  - `20260213220000_add_usage_table.sql`
    - Création de `usage` avec RLS activée + policies pour SELECT/INSERT uniquement par l’utilisateur (`auth.uid() = user_id`).
  - `20260213230000_add_plan_to_profiles.sql`
    - Ajout de la colonne `plan` sur `profiles` avec check constraint (`IN ('free', 'pro')`) + index sur `plan`.
  - Autres migrations (non toutes détaillées ici) : ajout d’email sur profiles, background style panel, blocks couleur, scenario chapters, etc.

### Edge Functions & sécurité

- **`generate-scenario-ai` (Groq)**
  - Vérification JWT :
    - Récupère `Authorization` depuis le header, et vérifie l’utilisateur en appelant `/auth/v1/user` avec une clé API (ANON ou service role selon disponibilité) avant de traiter la requête.
    - Refuse requêtes sans header ou avec JWT invalide/expiré (`401`).
  - Mode `panels` :
    - Nettoie la sortie du LLM, tente de fermer un JSON tronqué, parse le JSON et renvoie `panels` (liste de descriptions + contexte).
    - En cas de JSON invalide, renvoie une erreur explicite plutôt que la réponse brute (bonne hygiène de sécurité).

- **`generate-asset-image` (FAL.ai + quotas)**
  - Vérification JWT stricte via `/auth/v1/user` (service key).
  - Récupère le **plan utilisateur** via `profiles.plan` (REST API Supabase) et applique des limites :
    - **Free** : max 20 générations/mois, pas d’images de référence, modèle Schnell.
    - **Pro** : 300 générations/mois, images de référence autorisées, modèles FLUX-2 Pro/Edit.
  - Vérifie le **quota** sur la table `usage` (compte exact sur le mois en cours). Au-delà du quota :
    - Renvoie 429 avec métadonnées (quota_exceeded, current_usage, max_usage, plan).
  - Vérifie que l’asset appartient à l’utilisateur avant de générer.
  - Télécharge l’image FAL, l’upload dans Storage (`dreamweave` bucket) avec chemin structuré par utilisateur/projet/asset, met à jour la bonne colonne d’URL dans la table `assets`, et enregistre l’usage.

- **`generate-panel-image`**
  - Vérification JWT via `/auth/v1/user`.
  - Vérifie que le panel appartient à l’utilisateur, puis retrouve `project_id` depuis `chapters`.
  - Construit un prompt enrichi avec :
    - Style projet (`style_template`).
    - Contexte du chapitre (`context_chapter`).
    - Noms des assets utilisés dans le bloc.
    - Instruction forte sur la taille exacte (width/height) et remplissage complet du cadre.
  - Génère l’image (Flux-2 Pro), l’upload dans Storage (`projects/{project_id}/panels/{panel_id}/blocks/{block_id}.png`), renvoie l’URL publique.

### Sécurité & RLS

- **Points positifs**
  - Toutes les fonctions critiques **re-vérifient** l’utilisateur à partir du JWT côté serveur.
  - Vérifications d’appartenance (asset/panel → user_id) avant opération.
  - RLS explicite sur la table `usage`.
  - Les clés sensibles (SERVICE_ROLE, FAL, GROQ) ne sont jamais utilisées côté front.

- **Zones d’ombre / dépendances externes**
  - Les policies RLS pour `projects`, `chapters`, `panels`, `assets`, `profiles` ne sont pas définies dans les migrations visibles (probable configuration via dashboard Supabase).
    - L’audit ne peut pas confirmer que seule `auth.uid()` accède aux données de l’utilisateur sur ces tables.
  - `verify_jwt = false` pour les Edge Functions repose entièrement sur la logique interne de chaque fonction :
    - Tant que toutes les fonctions valident bien `Authorization` et refusent les requêtes sans JWT, c’est OK.
    - Un oubli dans une nouvelle fonction ou une future modification pourrait exposer une surface d’attaque.

## Risques / points de vigilance techniques

- **Sécurité**
  - Edge Functions avec `verify_jwt = false` :
    - Actuellement bien sécurisées via `Authorization` + appel `/auth/v1/user`.
    - Risque : un futur ajout de fonction ou modification qui oublierait cette étape ouvrirait une brèche.
  - RLS non visible pour les tables principales :
    - L’audit ne peut pas confirmer que seules les données de l’utilisateur sont visibles côté client.
    - Il faut vérifier la config RLS Supabase directement dans le dashboard.

- **Quotas IA & coûts**
  - Rate limiting géré via la table `usage` + Edge Function `generate-asset-image`.
  - En cas de dysfonctionnement (erreur lors de l’écriture dans `usage`, policy incorrecte, etc.), le quota pourrait ne pas s’appliquer, créant un risque de dépassement de coûts IA.

- **Robustesse**
  - Très peu de tests, en particulier :
    - Aucun test sur la logique de construction de prompts IA (scénario/panels/assets).
    - Aucun test sur la logique de quotas et de sélection de modèle.
    - Aucun test E2E pour les flows critiques (auth, création de projet, génération d’assets).
  - Pas de CI visible : les régressions peuvent passer en prod sans être détectées automatiquement.

- **Qualité/maintenabilité**
  - Règles TS et ESLint relativement permissives sur certains points (null, any implicites possibles, unused vars).
  - Duplications légères entre config de plans côté front (`TIER_CONFIG`) et côté Edge Function (`TIER_LIMITS`).

## 1. Architecture & Structure

### Points forts
- Séparation claire : `pages/` → `components/` → `hooks/` → `services/` → `integrations/`
- Types centralisés dans `src/types/index.ts` dérivés du schéma Supabase — bonne source de vérité unique
- Lazy loading de toutes les pages dans `App.tsx` avec `Suspense` + `PageLoader` — bon pour le bundle
- React Query correctement configuré (singleton `queryClient` hors composant, `staleTime` raisonnable)
- Auth PKCE + session persistante dans `localStorage` via `supabase/client.ts`

### Problèmes identifiés
- **`ChapterDetail.tsx` : 2 611 lignes** — God Component qui gère l'intégralité de l'éditeur de panels (drag, resize, bulles, couleurs, scénario, IA). Impossible à maintenir, tester ou refactoriser sans risque.
- La route `/dashboard/projects/new` (App.tsx:74) rend le composant `Projects` — pas de page dédiée à la création, logique de création mélangée dans la liste.
- `EditionSection.tsx` reçoit un prop `project: Project` qu'il renomme immédiatement `_project` (ligne 55) et n'utilise jamais — prop inutile dans l'interface.

## 2. Code Quality & TypeScript

### Bug TypeScript critique
**`CharacterView` n'est pas défini dans `src/types/index.ts`** mais est importé dans `src/hooks/useAssetGeneration.ts:7` :
```ts
import type { Asset, CharacterView, Project, UserPlan } from "@/types";
```
Ce type n'est exporté nulle part. `AssetLibrary.tsx` utilise l'union inline `"profile_left" | "profile_right" | "back"` à la place. Le build TypeScript en mode strict échoue sur ce point.

### Duplication de code
- **`emailRegex`** est copiée-collée identiquement dans 3 fonctions de `useAuth.tsx` : `signUp`, `signIn`, `resetPassword` (lignes 58, 132, 216). Doit être extraite en constante de module.
- **`BLOCK_PRESETS` et `COLOR_BLOCK_PRESETS`** dans `services/panels.ts` (lignes 29–47) sont des tableaux identiques — inutilement dupliqués.
- `use-toast.ts` existe à la fois dans `src/hooks/` et `src/components/ui/` — risque de divergence.

### Nommage incohérent
- Les fichiers hooks alternent entre `useXxx.ts` et `use-xxx.tsx` sans règle cohérente (`use-mobile.tsx`, `use-toast.ts` vs `useAuth.tsx`, `useProjects.ts`).
- Les fichiers d'audit anciens utilisent des formats de date différents (`Audit 13-02-2026.md`, `AUDIT-016-04-26.md`) — pas de standard.

### Complexité excessive
- `ChapterDetail.tsx` déclare **30+ états `useState`** et **12+ refs** dans le même composant. Aucun `useReducer` pour regrouper les états liés (ex. drag state, resize state).
- La logique de drag-and-drop est entièrement inline dans la page, sans abstraction ni hook dédié.

## 3. Performance

### Points forts
- Toutes les pages sont lazy-loadées (`React.lazy`).
- `queryClient` est un singleton avec `staleTime: 30_000` global.
- Le drag utilise des `ref` pour les mises à jour temps réel (pas de re-render pendant le drag) — bon pattern.

### Risques
- **`ChapterDetail.tsx`** : 30+ états peuvent déclencher des re-renders en cascade. Sans `useMemo` ou `useCallback` sur les handlers les plus lourds (génération d'image par bloc), des renders inutiles sont probables.
- `useAssets`, `useProject`, `useScenarioChapters`, `usePanels` sont tous appelés dans `ChapterDetail` — 4 requêtes parallèles au mount, mais sans coordination si l'une échoue.
- `estimatePanelCount()` dans `panels.ts:82` recalcule à chaque rendu si appelé dans JSX sans `useMemo`.

## 4. Sécurité

### Points forts
- PKCE flow (`flowType: "pkce"`) — protection CSRF sur OAuth.
- Validation email côté hook avant appel Supabase (signUp, signIn, resetPassword).
- Refresh de session systématique avant chaque appel à une Edge Function.
- `VITE_SUPABASE_PUBLISHABLE_KEY` correctement utilisé comme clé anonyme publique (non secrète).

### Risques
- **`updateUserPlan` côté client** (`useUserPlan.ts:45`) : un utilisateur peut appeler `changePlan.mutate("pro")` directement depuis la console du navigateur et upgrader son plan sans paiement, si Supabase RLS autorise la mise à jour de `profiles.plan` par l'utilisateur lui-même. Vérifier que la RLS restreint cette mise à jour à des fonctions Edge Function authentifiées (webhook Stripe, etc.).
- **`replacePanelsFromOutline`** (`services/panels.ts:175–183`) : supprime TOUS les panels du chapitre puis recrée. Si la création échoue à mi-chemin, les données sont partiellement perdues — pas de transaction atomique côté Supabase.
- Le header `apikey` dans `generatePanelBlockImage` (`panels.ts:228`) envoie `VITE_SUPABASE_PUBLISHABLE_KEY` en clair — c'est la clé anonyme Supabase, normale pour un client frontend. Vérifier que les Edge Functions valident bien le JWT et non seulement l'apikey.

## 5. UX & Accessibilité

### Points forts
- États de chargement (skeletons `animate-pulse`) sur les listes de chapitres et projets.
- Dialogues de confirmation (`AlertDialog`) avant toute suppression destructive.
- Toast errors avec `variant: "destructive"` sur tous les appels API.

### Lacunes
- **Accessibilité** : les boutons icône-seulement (ChevronUp/Down dans `EditionSection.tsx`, bouton Pencil dans `ProjectDetail.tsx`) ont un `title` mais pas d'`aria-label` — les lecteurs d'écran ne liront pas le `title` HTML.
- Sidebar fixe dans `ProjectDetail.tsx` (`fixed left-4 top-24 w-[260px]`) : sur des écrans entre 1024px et 1280px, le contenu principal n'a pas de `margin-left` correspondant — risque de recouvrement visuel.
- `panelEditorRightTool` dans `ChapterDetail` est déclaré avec le type `"chapter-text"` fixe — state inutile si la valeur ne change jamais.
- Aucune gestion du cas où `projectId` ou `chapterId` est `undefined` dans les appels de hooks de `ChapterDetail` (ex. `useCreatePanel(chapterId ?? "")`) — une chaîne vide sera envoyée à Supabase.

## Résumé exécutif

Le projet est structurellement sain (séparation services/hooks/pages, types dérivés du schéma Supabase, auth robuste). Les deux points critiques à traiter immédiatement sont :
1. Le type `CharacterView` manquant qui provoque une erreur TypeScript.
2. La protection serveur des upgrades de plan utilisateur.

Le chantier principal à moyen terme est le refactoring de `ChapterDetail.tsx` — actuellement ingérable à 2 600 lignes — qui sera nécessaire avant tout ajout de feature sur l'éditeur de panels.

# Audit technique — 15/04/2026

> Audit global du dépôt `dream-canvas` (frontend, hooks/services, edge functions, docs).  
> Axes: sécurité, performance, duplication, robustesse, cohérence documentaire.

---

## 1) Résumé exécutif

Le projet est fonctionnel et riche, mais présente plusieurs risques structurels:

- une vulnérabilité **business critique** (changement de plan `free -> pro` côté client),
- un risque de fuite de tokens en logs dans le flow reset password,
- des points de sécurité backend à durcir (interpolation de paramètres dans requêtes service-role, CORS permissif),
- des hotspots de performance/maintenabilité (composants monolithiques),
- une duplication de codebase (`dream-canvas/dream-canvas`) qui augmente fortement le risque de divergence.

---

## 2) Findings critiques (High)

### H1 — Escalade de plan utilisateur (Free vers Pro)

- **Fichiers**
  - `src/pages/Plans.tsx`
  - `src/hooks/useUserPlan.ts`
  - `supabase/migrations/20260209195229_f35fa33d-369d-4bfe-b797-2c3227a88c61.sql`
- **Constat**
  - Le plan est modifié directement par le client (`profiles.plan`) et la policy actuelle autorise l’update du profil utilisateur.
- **Risque**
  - Contournement du paywall / perte de revenus / abus des quotas Pro.
- **Action recommandée**
  - Retirer l’update `plan` côté client.
  - Mettre le changement de plan uniquement via backend (webhook paiement + service role).
  - Bloquer SQL toute mise à jour de `plan` hors flux serveur contrôlé.

### H2 — Fuite potentielle de tokens en console

- **Fichier**
  - `src/pages/ResetPassword.tsx`
- **Constat**
  - Plusieurs `console.log` exposent `access_token` / `refresh_token` durant la réinitialisation.
- **Risque**
  - Exposition de session si console partagée / capture / extension malveillante.
- **Action recommandée**
  - Supprimer immédiatement les logs sensibles.
  - Conserver au besoin uniquement des logs non sensibles (état/étape).
- **Statut**
  - ✅ Corrigé le 15/04/2026: suppression des `console.log` du flow de reset contenant ou manipulant des tokens/session.

### H3 — Duplication de codebase active

- **Constat**
  - Présence de deux arbres actifs:
    - `dream-canvas/src/...`
    - `dream-canvas/dream-canvas/src/...`
  - Ex confirmé sur `ChapterDetail.tsx`.
- **Risque**
  - Correctifs appliqués au mauvais endroit, comportement différent selon build/outils, dette de maintenance majeure.
- **Action recommandée**
  - Définir immédiatement une base canonique.
  - Archiver/supprimer l’arbre dupliqué.
  - Ajouter contrôle CI pour empêcher une seconde arborescence applicative.
- **Statut**
  - ✅ Corrigé le 15/04/2026: suppression des 15 fichiers du miroir partiel `dream-canvas/dream-canvas/`.
  - Base canonique confirmée: `dream-canvas/`.

## 3) Findings importants (Medium)

### M1 — `setState` pendant le render dans `ProjectDetail`

- **Fichier**
  - `src/pages/ProjectDetail.tsx`
- **Constat**
  - `setStyleTemplate` et `setStyleInitialized` sont appelés dans le corps du composant.
- **Risque**
  - Re-renders parasites / comportement fragile en Strict Mode.
- **Action**
  - Déplacer cette initialisation dans un `useEffect`.
- **Statut**
  - ✅ Corrigé le 15/04/2026: suppression du `setState` dans le render, synchronisation déplacée dans un `useEffect`.

### M2 — Interpolation non validée dans requêtes PostgREST service-role

- **Fichiers**
  - `supabase/functions/generate-asset-image/index.ts`
  - `supabase/functions/generate-panel-image/index.ts`
- **Constat**
  - `asset_id` / `panel_id` injectés directement dans l’URL PostgREST.
- **Risque**
  - Surface d’attaque inutile sur endpoints sensibles.
- **Action**
  - Validation stricte UUID avant usage + `encodeURIComponent`.
  - Idéalement passer par client supabase server-side typé au lieu d’URL interpolées.
- **Statut**
  - ✅ Corrigé le 15/04/2026: validation UUID stricte ajoutée + encodage `encodeURIComponent` sur les IDs interpolés.

### M3 — CORS permissif par défaut (`*`)

- **Fichiers**
  - Edge functions (`generate-asset-image`, `generate-panel-image`, `generate-scenario-ai`, `generate-style-template-images`)
- **Risque**
  - Surface cross-origin élargie.
- **Action**
  - Rendre `ALLOWED_ORIGIN` obligatoire en prod, sans fallback wildcard.
- **Statut**
  - ✅ Corrigé le 15/04/2026: suppression du fallback `*` et erreur explicite `500` si `ALLOWED_ORIGIN` n’est pas configurée.

### M4 — `verify_jwt=false` sur toutes les edge functions

- **Fichier**
  - `supabase/config.toml`
- **Risque**
  - Dépendance totale à la vérification custom (erreur future = exposition).
- **Action**
  - Réactiver `verify_jwt=true` pour les fonctions user-facing quand possible.
  - Garder exceptions uniquement pour cas M2M strictement contrôlés.
- **Statut**
  - ✅ Corrigé le 15/04/2026: `verify_jwt=true` réactivé pour les fonctions user-facing.
  - Exception maintenue pour `generate-style-template-images` (endpoint admin/script avec token technique).

### M5 — Hotspots monolithiques et coût de maintenance

- **Fichiers**
  - `src/pages/ChapterDetail.tsx`
  - `src/components/project/ScenarioSection.tsx`
- **Risque**
  - Régressions fréquentes, faible testabilité, optimisation difficile.
- **Action**
  - Découper en sous-composants + hooks métier dédiés.
- **Statut**
  - ✅ Corrigé (itération 1) le 15/04/2026:
    - extraction d’une logique de fermeture/réinitialisation dans `ChapterDetail`,
    - factorisation du déclenchement IA chapitre dans `ScenarioSection`.
  - ℹ️ Refactor structurel complet (découpage massif en sous-composants/hooks) reste recommandé en phase 2.

### M6 — Invalidation réseau potentiellement excessive

- **Fichiers**
  - `src/pages/ChapterDetail.tsx`
  - `src/hooks/usePanels.ts`
- **Risque**
  - Trop de refetchs sous interactions fréquentes (drag/resize/save).
- **Action**
  - Standardiser une stratégie mutation (optimistic update + invalidation ciblée).
- **Statut**
  - ✅ Corrigé (itération 1) le 15/04/2026:
    - suppression des `refetchQueries` redondants dans `ChapterDetail` après `updatePanelMutation`.
    - conservation des mises à jour optimistes + rollback + invalidation gérée par le hook `useUpdatePanel`.
  - ℹ️ Optimisation plus avancée possible ensuite (centralisation complète de la stratégie optimistic/invalidation).

### L2 — Rigueur statique TypeScript/ESLint partiellement relâchée

- **Constat**
  - certaines règles peu strictes (unused vars / null safety).
- **Action**
  - resserrer progressivement les règles (d’abord warnings, puis errors).
- **Statut**
  - ✅ Corrigé le 15/04/2026 (phase 1 progressive):
    - activation de `@typescript-eslint/no-unused-vars` en niveau `warn`,
    - ajout d’une convention explicite `_` (`args/vars/caughtErrors`) pour les cas intentionnels.

## 5) Plan d’action recommandé (ordre)

1. **Bloquer l’escalade de plan** (`profiles.plan` serveur-only).  
2. **Supprimer logs tokens** (`ResetPassword.tsx`).  
3. **Résoudre la duplication de codebase** (`dream-canvas/dream-canvas`).  
4. **Durcir Edge Functions** (UUID validation, CORS, `verify_jwt`).  
5. **Corriger `ProjectDetail` setState dans render**.  
6. **Refactor progressif des monolithes** (`ChapterDetail`, `ScenarioSection`).  
7. **Nettoyage docs + durcissement lint/TS**.

- Vérifications dynamiques recommandées ensuite:
  - tests e2e auth + changement de plan,
  - tests charge légère sur édition panel,
  - revue RLS/policies en environnement staging.
